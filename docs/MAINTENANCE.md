# 维护规范

这份规范用于保证整本手册的结构一致、信息可追溯、后续更新成本可控。

## 资料来源规则

- 只收录官方来源。
- 优先级从高到低：
  1. 官方 API Reference
  2. 官方 Guides / Docs
  3. 官方 SDK 仓库
  4. 官方博客 / Changelog / Release Notes
- 非官方博客、论坛帖子、第三方聚合页不作为事实依据，只能作为线索。

## 页面状态标签

每个页面开头建议标注以下字段：

- 收录状态：`待补充` / `整理中` / `已完成`
- 优先级：`高` / `中` / `低`
- 最近校验日期：`YYYY-MM-DD`

## 厂商页最低要求

至少补齐以下内容后，页面才算进入 `整理中`：

- 官网
- 官方文档入口
- API 基础地址
- 鉴权方式
- 核心接口名称
- 工具调用或结构化输出说明
- 流式输出说明

## 协议页最低要求

至少补齐以下内容后，页面才算进入 `整理中`：

- 官方规范地址
- 协议解决的问题
- 核心对象或核心字段
- 一条典型交互流程
- 与主流厂商或主流接口的映射关系

## 摘要写法

- 先给结论，再给原始链接。
- 优先做结构化摘要，不直接复制原文段落。
- 对“兼容”一词保持谨慎，明确说明兼容的是接口路径、字段、行为，还是 SDK 用法。
- 对存在版本差异的内容，明确写出版本或日期。

## 链接组织建议

建议每个页面都保留一个“官方资料索引”区块，至少包含：

- 文档主页
- API Reference
- SDK
- Changelog / Release Notes
- 最佳实践或迁移指南

## 推荐填充顺序

1. 补官方入口链接。
2. 补请求地址、鉴权和模型字段。
3. 补工具调用、流式、多模态。
4. 最后补限流、错误码、计费和迁移说明。

## 命名建议

- 厂商目录名用英文或拼音短横线形式。
- 页面标题保留官方品牌写法。
- 同一术语在全仓库保持一致，例如：
  - `工具调用`
  - `流式输出`
  - `多模态`
  - `结构化输出`
  - `OpenAI Compatible API`
