# OpenAI Compatible API

## 一句话定义

所谓 OpenAI Compatible API，并不是一份真正被统一制定、统一治理的正式标准，它更像一种事实上的生态约定：大量平台通过兼容 OpenAI 风格接口，来降低接入门槛与迁移成本。

## 官方来源

- 事实参考来源：<https://platform.openai.com/docs/api-reference>
- 常见兼容目标：`/v1/chat/completions`、`/v1/embeddings` 及部分周边接口
- 生态背景说明：应结合各厂商官方兼容文档理解，而不应把任意第三方整理当作标准文本

## 核心对象

- 参与角色：API Client、模型服务平台、兼容层适配实现
- 关键对象：model、messages、tools、response_format、stream 等常见字段
- 核心能力：在尽量保持调用形状不变的前提下，让应用以较低成本切换到底层不同模型平台

## 关键流程

- 初始化：通过 API Key、Base URL 与模型名称完成最小接入配置
- 请求 / 响应：多数兼容实现围绕 chat completions、embeddings 等接口提供相似请求体
- 工具调用：部分字段名与事件形态表面兼容，但细节行为往往存在偏差
- 流式交互：通常借用 SSE 风格返回，但 chunk 结构、结束事件与增量字段并不总是完全一致
- 错误处理：错误码、限流语义、字段严格度与失败行为常常是兼容层最容易暴露差异的地方

## 与其他协议的边界

- 与 OpenAI 官方 API：兼容 API 以其为事实参照，但不等于具备完全同等的行为语义与能力覆盖
- 与 MCP：OpenAI Compatible API 关注的是模型推理接口形状兼容，MCP 关注的是模型运行时如何接入工具与外部上下文
- 与 A2A：A2A 关注 Agent 之间的任务与能力协作，不属于模型推理接口兼容问题
- 与厂商私有协议：很多厂商会同时提供兼容接口与原生接口，后者通常能暴露更完整的能力边界

## 实践中的意义

- 适用场景：快速接入替代模型、做多厂商切换、构建统一 SDK 封装、验证基础可用性
- 不适用场景：需要依赖某家平台的独有能力、精细流式事件、复杂工具调用语义或最新多模态特性时
- 最容易误判的地方：把“请求能发通”当成“语义完全兼容”；事实上，真正的差异往往出现在工具调用、流式、结构化输出、错误处理与边界能力上

## 常见不兼容行为

- 模型名称与能力矩阵不一致
- `system`、`developer`、`user` 等消息语义支持不一致
- tool calling 字段表面相似，但执行轮次与返回形态不同
- stream chunk 结构、结束事件与 finish reason 表达不同
- JSON schema / structured output 支持程度不同

## 备注

如果你的目标只是“先跑起来”，兼容接口往往足够；如果你的目标是“稳定迁移”或“能力对齐”，最终仍然要回到各厂商原生文档核对真实边界。
