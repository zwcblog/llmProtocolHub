# 流式输出格式

## 一句话定义

流式输出解决的不是“模型更聪明”这个问题，而是“结果如何更早、更连续地到达调用方”。它表面上像传输细节，实际上会直接影响前端体验、工具调用过程、延迟感知与兼容层实现复杂度。

## 官方来源

- OpenAI 参考：<https://platform.openai.com/docs/api-reference/streaming>
- Anthropic 参考：<https://docs.anthropic.com/en/api/messages-streaming>

## 核心对象

- 参与角色：模型服务端、客户端消费方、可选的中间兼容层
- 关键对象：事件流、增量片段、结束标记、错误事件、可选的工具调用增量信息
- 核心能力：在完整响应结束前，持续向调用方发送中间结果

## 常见传输形态

- SSE：当前最常见，也最容易被 OpenAI 风格兼容层复用
- Chunked HTTP Response：常见于底层实现视角
- WebSocket / Realtime：更适合低延迟双向交互，但不等于所有“流式输出”都需要 WebSocket

## 关键流程

- 请求发起：客户端显式声明需要 stream 模式
- 增量返回：服务端持续发送文本片段、结构化字段或事件对象
- 中间事件：某些平台会在流中穿插 tool call、usage、message start / stop 等事件
- 结束确认：通过 finish reason、done 事件或连接结束表明流完成
- 错误处理：客户端需要区分“正常结束”“中途失败”“格式异常”“上游断流”等不同情况

## 厂商差异

- 事件粒度不同：有的平台按 token / delta 粒度，有的平台按消息块或事件粒度
- 结束语义不同：`[DONE]`、事件对象、连接关闭等方式并存
- 工具调用流式表达不同：部分平台会增量返回调用参数，部分平台更偏完整事件
- usage / metadata 返回时机不同：有的在末尾统一返回，有的在单独事件中返回

## 与其他协议的边界

- 与工具调用：流式只是承载方式之一，工具调用语义本身独立存在
- 与 Realtime：Realtime 通常是更高阶、双向、低延迟的交互模型；流式输出可以只是单向增量返回
- 与兼容层：很多兼容问题并不出在主请求体，而是出在流式事件形状与结束语义

## 实践中的意义

- 适用场景：聊天 UI、逐步渲染、长文本生成、工具调用链过程可视化、降低首 token 体感延迟
- 不适用场景：只关心最终结构化结果、对中间过程无感知需求的离线批处理任务
- 最容易误判的地方：把“能 stream 出文本”当成“流式语义完全兼容”；真实差异往往藏在事件格式、结束条件与异常处理上

## 备注

如果你在做统一封装，流式支持通常是最容易暴露厂商差异的一层；真正难的不是把连接打开，而是把事件语义收敛成稳定的上层抽象。
