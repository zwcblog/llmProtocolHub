# LLM Protocol Hub

一个面向 AI 厂商与协议的**官方入口索引站**。

它不试图重写各家平台文档，也不维护一套注定会迅速过时的参数摘录。这个项目只做一件更朴素、也更有长期价值的事：

> 把真正值得抵达的官方入口，重新按统一结构整理出来。

在今天的 AI 生态里，信息并不稀缺，稀缺的是**准确定位原始出处的能力**。

同一个平台的资料，常常散落在：

- 官网
- 开发者文档
- API Reference
- SDK / GitHub
- 控制台 / 平台入口
- Changelog / Release Notes

当你真正开始接入某个模型平台，或者比较多个协议实现时，最花时间的往往不是“理解 API”，而是先弄清楚：**应该先看哪里，哪里才是权威入口，哪些页面只是外围说明。**

LLM Protocol Hub 试图解决的，正是这个问题。

---

## 站点定位

这个项目更接近：

- AI 厂商官方入口导航
- 协议规范索引
- 面向开发者的资料编目站

而不是：

- 二次改写的 API 教程
- 大而全的 LLM 百科
- 长篇参数说明书
- 替代官网的镜像文档

这里提供的是**索引、边界和最少量的结构化提示**。
具体字段、接口行为、最新版本与兼容细节，仍应以各自官方文档为准。

## 内容结构

站点主要由两部分组成：

### 1. 厂商卷

按平台整理官方入口，通常包含：

- 官网
- 官方文档
- API Reference
- SDK / GitHub
- Changelog / Release Notes
- Console / Platform

必要时补充极短的结构化说明，例如：

- 主 API 形态
- 兼容层情况
- 工具调用 / 流式 / 多模态入口
- 常见迁移边界

### 2. 协议卷

按协议整理官方规范与参考实现，通常包含：

- 官方规范
- 核心对象
- 关键流程
- 参考实现
- 与相邻协议的边界

## 维护原则

这个项目遵循以下原则：

- 只收录官方来源
- 优先链接，不堆砌解释
- 用统一结构降低检索成本
- 尽量减少二次转述
- 页面目标是“快速抵达官网”，而不是“替代官网”

---

## 适合谁

如果你符合下面任一场景，这个站点会比一份普通收藏夹更有用：

- 需要快速找到某家模型平台的官方入口
- 想确认某个协议的权威规范与参考实现
- 在多个厂商之间做接入、迁移或对照
- 不想先读一堆二手文章，只想尽快回到原始来源

---

## 不适合谁

如果你想要的是：

- 从零开始的教程
- 逐字段参数解释
- 全量兼容性手册
- 深度实战案例合集

那这里可能不是最合适的地方。

这个站点更像目录，而不是正文。
